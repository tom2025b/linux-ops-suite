Sidecar loop — risk tagging, via Workstate only

Workstate snapshot findings (from Bulwark)⟶ Toolbox-Bridge toolbox-bridge.workstate-feed.v1⟶ ScriptVault sidecar metadata

Toolbox-Bridge (Rust, in the umbrella workspace) reads Bulwark’s findings from the compiled snapshot.json — never from Bulwark directly — and publishes risk:*/owner:* tags plus badged descriptions as a versioned feed in Workstate’s feeds directory (feeds/toolbox-bridge.json). ScriptVault consumes that feed and merges it with its usual sidecar-wins rules. No tool talks to another tool directly; every hop is a Workstate artifact.

Legend

  • Producer — own repo, read-only, emits a feed
  • Compiler — the only thing that reads feeds
  • Consumer — reads the snapshot only
  • Sidecar loop — snapshot-derived risk-tag path

Rules this encodes

  • One-way, file-based contracts — no tool imports another’s code; JSON only.
  • Shapes pinned by JSON Schemas in the umbrella contracts/ dir, validated in CI.
  • Workstate is the single compiler — fans in 3, fans out 1.
  • RexOps only ever reads snapshot.json.

Note: feed names and the snapshot version verified against the 9 schemas in contracts/ and docs/INTEGRATION_MAP.md. ToolFoundry, Bulwark, Proto, and Toolbox-Bridge emit a workstate-feed; ScriptVault’s export is RexOps-bound. The snapshot→Bridge→ScriptVault path is an indirect loop by design — mediated entirely by Workstate artifacts.
